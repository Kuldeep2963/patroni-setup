01 Initial server preparation
ℹ Make sure you have root or sudo access to a fresh Ubuntu / Debian server with internet connectivity before starting.
SSH into server ssh root@server_ip
▼
ssh root@server_ip

ssh opens a secure encrypted connection to your remote server. Replace server_ip with your actual IP address or hostname. This logs you in as the root superuser.

Create a new user adduser user_name
▼
adduser user_name

Creates a new system user account with a home directory. It will prompt you to set a password and fill in some optional details. Replace user_name with something meaningful like dbadmin or your company name. It is a security best practice to avoid working as root for day-to-day tasks.

Grant sudo access usermod -aG sudo user_name
▼
usermod -aG sudo user_name

Gives your new user administrative privileges. -a appends without removing from other groups. -G sudo adds the user to the sudo group, which allows running commands as administrator.

02 Install required software
Install Docker sudo apt install docker.io -y
▼
sudo apt install docker.io -y

Docker is the container runtime that will run Patroni, PostgreSQL, etcd, and HAProxy as isolated services. apt install downloads and installs the package from Ubuntu's official repository. -y skips the confirmation prompt.

Install Git sudo apt install git -y
▼
sudo apt install git -y

Git is a version control tool used to download (clone) the Patroni source code and configuration files from GitHub.

Add user to Docker group sudo usermod -aG docker user_name
▼
sudo usermod -aG docker user_name

By default only root can run Docker commands. This adds your user to the docker group so you can use Docker without typing sudo every time.

Apply group immediately newgrp docker
▼
newgrp docker

Activates the new group membership in your current shell session. Without this you would need to log out and log back in for the Docker group change to take effect.

Install Docker Compose plugin sudo apt install docker-compose-plugin -y
▼
sudo apt install docker-compose-plugin -y

Installs Docker Compose V2 as a Docker CLI plugin. This setup uses docker compose (V2 — space, no hyphen) instead of the legacy standalone docker-compose binary. V2 is the current standard, faster, and built directly into the Docker CLI. Verify with: docker compose version

03 Clone and configure the Patroni repository
Clone Patroni repo git clone https://github.com/patroni/patroni.git
▼
git clone https://github.com/patroni/patroni.git

Downloads the complete official Patroni repository from GitHub to your server — including all Docker Compose files, Dockerfiles, and configuration templates.

Enter the folder cd patroni
▼
cd patroni

Navigates into the downloaded Patroni directory. All subsequent commands should be run from inside this folder.

⚠ Open docker-compose.yml and docker-compose-citus.yml and replace every occurrence of demo with your project name. Also update Dockerfile.citus: change PG_MAJOR to PG_MAJOR-bookworm for better SSL and OS compatibility.
ℹ Follow all remaining steps in the official Docker README: github.com/patroni/patroni/blob/master/docker/README.md
04 Update Patroni configuration (HAProxy / Coordinators)

After the containers are running, update the cluster configuration via patronictl edit-config to enable replication, SSL, and performance tuning.

Edit cluster config docker exec -it demo-haproxy patronictl edit-config
▼
docker exec -it demo-haproxy patronictl edit-config

docker exec -it opens an interactive terminal session inside a running container. patronictl edit-config opens the Patroni cluster configuration in a text editor inside the container. Paste in the YAML block shown below.

patroni-config.yaml — paste this into the editor 
loop_wait: 10
maximum_lag_on_failover: 1048576

postgresql:
  parameters:
    checkpoint_completion_target: 0.9
    checkpoint_timeout: 30min
    citus.node_conninfo: "sslrootcert=/etc/ssl/certs/ssl-cert-snakeoil.pem sslkey=/etc/ssl/private/ssl-cert-snakeoil.key sslcert=/etc/ssl/certs/ssl-cert-snakeoil.pem sslmode=verify-ca"
    effective_cache_size: 61GB
    effective_io_concurrency: 200
    maintenance_work_mem: 2GB
    max_connections: 1000
    max_parallel_workers_per_gather: 0
    random_page_cost: 1.1
    shared_buffers: 36GB
    ssl: 'on'
    ssl_ca_file: /etc/ssl/certs/ssl-cert-snakeoil.pem
    ssl_cert_file: /etc/ssl/certs/ssl-cert-snakeoil.pem
    ssl_key_file: /etc/ssl/private/ssl-cert-snakeoil.key
    synchronous_commit: false
    wal_buffers: 32MB
    wal_level: replica
    work_mem: 32MB

  pg_hba:
    - local all all trust
    - hostssl replication replicator all md5 clientcert=verify-ca
    - hostssl all all all md5 clientcert=verify-ca

  use_pg_rewind: true

retry_timeout: 10
synchronous_mode: quorum
ttl: 30
Restart coordinators docker restart demo-coord1 demo-coord2 demo-coord3
▼
docker restart demo-coord1 demo-coord2 demo-coord3

Restarts all three Citus coordinator containers so the updated Patroni configuration takes effect. Coordinators manage query routing in a Citus distributed cluster.

Restart workers docker restart demo-work1-1 demo-work1-2 demo-work2-1 demo-work2-2
▼
docker restart demo-work1-1 demo-work1-2 demo-work2-1 demo-work2-2

Restarts the Citus worker containers that store the actual sharded table data. Adapt the container names if you have more or fewer worker groups.

06 Verify the PostgreSQL parameters
Check shared_buffers docker exec -it demo-coord1 psql -U postgres -c "SHOW shared_buffers;"
▼
docker exec -it demo-coord1 psql -U postgres -c "SHOW shared_buffers;"

psql -c runs a single SQL command and exits. SHOW prints the live value of any PostgreSQL configuration parameter — here confirming that shared_buffers was applied correctly after the restart.

Check max_connections docker exec -it demo-coord1 psql -U postgres -c "SHOW max_connections;"
▼
docker exec -it demo-coord1 psql -U postgres -c "SHOW max_connections;"

Confirms that the connection limit was applied correctly. You can use this same pattern with any parameter name from the configuration block to verify it.

07 SSL certificate setup and verification

After the cluster is running, locate the SSL certificates inside the containers, copy them to your host, fix their permissions, then test the SSL connection. Run all commands from your ~/patroni folder.

Find certificate in container docker exec -it demo-coord1 ls -la /etc/ssl/certs/ssl-cert-snakeoil.pem
▼
docker exec -it demo-coord1 ls -la /etc/ssl/certs/ssl-cert-snakeoil.pem

Confirms that the SSL certificate (.pem) exists inside the coordinator container at the expected path. You should see a file owned by root with permissions -rw-r--r--.

Find private key in container docker exec -it demo-coord1 ls -la /etc/ssl/private/ssl-cert-snakeoil.key
▼
docker exec -it demo-coord1 ls -la /etc/ssl/private/ssl-cert-snakeoil.key

Confirms that the private key (.key) exists inside the container. You should see a file with permissions -rw-r----- owned by root.

Copy certificate to host sudo docker cp demo-coord1:/etc/ssl/certs/ssl-cert-snakeoil.pem ~/patroni/ssl-cert-snakeoil.pem
▼
sudo docker cp demo-coord1:/etc/ssl/certs/ssl-cert-snakeoil.pem ~/patroni/ssl-cert-snakeoil.pem

docker cp copies a file from inside a running container to the host filesystem. This extracts the public certificate from the container into your ~/patroni/ folder so you can use it for client connections.

Copy private key to host sudo docker cp demo-coord1:/etc/ssl/private/ssl-cert-snakeoil.key ~/patroni/ssl-cert-snakeoil.key
▼
sudo docker cp demo-coord1:/etc/ssl/private/ssl-cert-snakeoil.key ~/patroni/ssl-cert-snakeoil.key

Copies the private key out of the container. Keep this file secure — it must never be shared or committed to version control.

Set certificate permissions chmod 644 ~/patroni/ssl-cert-snakeoil.pem
▼
chmod 644 ~/patroni/ssl-cert-snakeoil.pem

Sets the certificate file to be readable by all users (644). Public certificates do not need to be secret, so world-readable is correct and expected by PostgreSQL.

Set private key permissions chmod 600 ~/patroni/ssl-cert-snakeoil.key
▼
chmod 600 ~/patroni/ssl-cert-snakeoil.key

Restricts the private key to owner-only access (600). PostgreSQL and psql will refuse to use a private key file that is readable by other users.

⚠ If you encounter a permission error when testing the connection, also run: sudo chown user_name:user_name ~/patroni/ssl-cert-snakeoil.* — replacing user_name with your Linux username — then reapply the chmod commands above.
Test SSL connection psql "host=localhost port=5000 ..."
▼
psql "host=localhost port=5000 dbname=postgres user=postgres sslmode=verify-ca sslcert=./ssl-cert-snakeoil.pem sslkey=./ssl-cert-snakeoil.key sslrootcert=./ssl-cert-snakeoil.pem"

Connects to the database via HAProxy (port 5000) using SSL client certificates. sslmode=verify-ca enforces SSL and validates the server certificate against the CA. Run this from inside the ~/patroni/ folder.

✓ If the command successfully opens a psql prompt, your Patroni cluster with SSL is fully operational. You are ready to connect your backend application.
Export .p12 for backend cd ~/patroni
▼
cd ~/patroni

Navigate to the patroni folder where the certificates are stored before generating the .p12 file.

Generate keyStore.p12 openssl pkcs12 -export -out keyStore.p12 ...
▼
openssl pkcs12 -export \ -out keyStore.p12 \ -inkey ssl-cert-snakeoil.key \ -in ssl-cert-snakeoil.pem \ -certfile ssl-cert-snakeoil.pem

Bundles the certificate and private key into a single .p12 (PKCS#12) file for use in Java or other backend applications. You will be prompted to set an export password — remember it, as it will be required every time the keystore is used. Copy keyStore.p12 to your backend server and configure it for SSL client authentication.

Verify keyStore.p12 was created ls -lh ~/patroni/keyStore.p12
▼
ls -lh ~/patroni/keyStore.p12

Confirms that the .p12 file was generated successfully. It should show a small non-zero file size (typically a few KB). A file size of 0 bytes indicates the export failed — re-run the openssl pkcs12 command and check for errors.

08 Configuration parameter reference
ParameterWhat it does
loop_wait: 10Patroni checks cluster health every 10 seconds.
maximum_lag_on_failoverMaximum bytes a replica can lag before being excluded from leader election during failover (1 MB).
shared_buffers: 36GBMemory dedicated to PostgreSQL for caching data pages. Typically set to ~25% of total RAM.
effective_cache_size: 61GBTells the query planner how much memory is available for caching (OS + PostgreSQL combined).
work_mem: 32MBMemory allocated per sort or hash operation within a single query.
maintenance_work_mem: 2GBMemory for maintenance tasks like VACUUM and CREATE INDEX.
wal_level: replicaEnables enough WAL detail to support streaming replication to standby nodes.
wal_buffers: 32MBMemory buffer for WAL data before it is flushed to disk.
synchronous_commit: falseTransactions do not wait for WAL flush — improves write performance at a small durability trade-off.
synchronous_mode: quorumPatroni requires a quorum of replicas to acknowledge writes before the primary commits them.
ssl: 'on'Enables SSL/TLS encryption for all database connections.
use_pg_rewind: trueAllows Patroni to resync a former primary with the new leader after failover without a full data copy.
ttl: 30Leader lock expiry in seconds — if the primary stops responding, a new election starts after this.
max_connections: 1000Maximum simultaneous client connections to PostgreSQL.
checkpoint_timeout: 30minHow often PostgreSQL flushes dirty pages to disk.
checkpoint_completion_target: 0.9Spreads checkpoint I/O over 90% of the interval to avoid disk spikes.
effective_io_concurrency: 200Concurrent I/O operations for query planning. Set higher for SSD storage.
random_page_cost: 1.1Tells the planner random reads are nearly as fast as sequential ones — optimal for SSDs.
09 pg_hba (host-based authentication) rules
RuleMeaning
local all all trustLocal Unix socket connections are trusted without a password — used for internal tooling.
hostssl replication replicator all md5 clientcert=verify-caReplica nodes must connect over SSL and present a valid client certificate signed by the CA.
hostssl all all all md5 clientcert=verify-caAll other connections must also use SSL and present a valid client certificate.
